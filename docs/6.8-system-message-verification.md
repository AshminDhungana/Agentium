# Task 6.8 — Verify system-message usage across both LLM APIs

**Status:** Verified — no defect found. System messages are correctly sent in
both the OpenAI-style and Anthropic-style provider paths.

## Before (the open question)

The project talks to two API families that model the system prompt
differently:

- **OpenAI-style** (Chat Completions): a message with `role: "system"`.
- **Anthropic** (Messages API): a dedicated top-level `system` field. Anthropic
  has **no** `system` *role* — putting the system text in a `role: "system"`
  message is invalid and is silently ignored by the API.

It was unclear whether Agentium actually transmitted an explicit system message
in each path, and whether its content was meaningful (ethos / constitution /
role) or empty.

## Investigation

Reviewed the request-building code in `backend/services/model_provider.py`:

| Path | Method | System message placement | Correct? |
|------|--------|--------------------------|----------|
| `OpenAICompatibleProvider.generate` | `model_provider.py:555` | `messages=[{"role": "system", "content": system_prompt}, {"role": "user", ...}]` (line 575) | ✅ |
| `OpenAICompatibleProvider.stream_generate` | `model_provider.py:629` | `{"role": "system", ...}` (line 648) | ✅ |
| `OpenAICompatibleProvider.generate_with_tools` | `model_provider.py:746` | `[{"role": "system", "content": system_prompt}] + conversation` (line 810) | ✅ |
| `AnthropicProvider.generate` | `model_provider.py:1071` | top-level `"system": system_prompt` (line 1091) | ✅ |
| `AnthropicProvider.stream_generate` | `model_provider.py:1134` | top-level `"system": system_prompt` (line 1153) | ✅ |
| `AnthropicProvider.generate_with_tools` | `model_provider.py:1167` | top-level `"system": system_prompt` (line 1230) | ✅ |
| `LocalProvider.generate` | `model_provider.py:1415` | folded into the user prompt (`f"{system_prompt}\n\nUser: {user_message}"`, line 1418) | ✅ by design |

`LocalProvider` deliberately folds the system prompt into the user turn because
many local endpoints (Ollama, llama.cpp) do not support a system role. This is
intentional and not a defect.

### System-prompt content

The system prompt is meaningfully constructed, not empty:

- `ModelService.generate_with_agent` builds it from the agent's ethos mission
  statement + behavioral rules (`model_provider.py:1590`).
- `ModelService.generate_with_agent_tools` builds it from the override (or
  ethos) and appends a hard response-length rule (`model_provider.py:1703`,
  `model_provider.py:1719`).
- Callers such as `AgentOrchestrator` (`agent_orchestrator.py:188`),
  `ChatService` (`chat_service.py:308`), and `LLMClient.decide`
  (`llm_client.py:504`) all supply a real system prompt.

So the "if not, determine what belongs in it" branch does **not** trigger.
Ethos, role, and behavioral rules are already present.

## After (verified state)

Both provider families send an explicit system message on every request path,
in the format each API expects:

- OpenAI-style → `role: "system"` message.
- Anthropic → top-level `system` field (never a `role: "system"` message).

This behavior is now locked in by regression tests in
`backend/tests/unit/test_provider_system_message.py`, which capture each
provider's outgoing request payload and assert an explicit system message is
present (OpenAI as a message, Anthropic as the top-level `system` field). The
tests fail if a future refactor drops or misplaces the system prompt.

## Conclusion

No code change required for system-message handling. The deliverable is the
verification above plus the regression test guarding the behavior.
